# Ралюма — презентация проекта

## Что сделано

**Система авторизации**
- Логин/пароль, JWT-токены, ролевая модель (админ/пользователь)
- Админ-панель управления пользователями

**Проекты**
- CRUD проектов: создание, редактирование, копирование, удаление
- Статусы производства, стеклопакетов, покраски
- Управление заказами (позиции заказа)

**Редактор секций**
- Полная форма СЛАЙД: рельсы, порог, фурнитура, профили, замки, ручки, щеколды
- Взаимозависимости полей (например, замок RS112/RS1002 — взаимоисключение)
- Системы: СЛАЙД, КНИЖКА, ЛИФТ, ЦС, КОМПЛЕКТАЦИЯ (формы для них)
- Нумерация секций, примечания, guard несохранённых изменений (Ctrl+S)

**SVG-схемы (в реальном времени)**
- Вид из помещения — рама, панели, стрелки сдвига, условные обозначения (ручки, замки, щеколды, глухие панели)
- Вид сверху — рельсы, панели, профили, межстекольный профиль

**Производственный лист (PDF)**
- Расчётный движок: ~750 строк формул (профили, стёкла, фурнитура, крепёж)
- Генерация A4 PDF через WeasyPrint
- Inline-редактирование (contenteditable) — можно поправить значение прямо в превью
- Сохранение правок в БД (document_overrides)
- Страница 2: доп. комплектующие с добавлением/удалением строк

**Инфраструктура**
- Деплой на VPS с auto-SSL (Caddy)
- Docker Compose (фронтенд + бэкенд)
- 130 автотестов

## Как это работает

```
Пользователь (браузер)
    │
    ▼
┌─────────────────┐    HTTPS (auto-SSL)
│   Caddy (proxy)  │◄── Caddy автоматически получает SSL-сертификат
│   + SPA-сервер   │
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
 /index.html   /api/*
 (React SPA)   (FastAPI)
    │              │
    │              ▼
    │         ┌──────────┐
    │         │ SQLite DB │  — проекты, секции, пользователи
    │         └──────────┘
    │              │
    │              ▼
    │         ┌──────────────┐
    │         │ slide_calc.py │  — расчёт профилей, стёкол, фурнитуры
    │         └──────┬───────┘
    │                │
    │                ▼
    │         ┌──────────────┐
    │         │  WeasyPrint   │  — HTML шаблон → A4 PDF
    │         └──────────────┘
    │
    ▼
Редактор секций ──► SVG-схемы обновляются в реальном времени
                     при изменении параметров формы
```

**Поток данных при редактировании:**
1. Пользователь меняет параметр в форме (ширина, профиль, ручка...)
2. React обновляет SVG-схемы мгновенно (без запроса к серверу)
3. При нажатии «Сохранить» (или Ctrl+S) — данные отправляются на бэкенд
4. При открытии «Производственный лист» — бэкенд пересчитывает всё через slide_calc.py и генерирует HTML-превью
5. Пользователь может поправить любое значение прямо в превью (contenteditable)
6. При скачивании PDF — WeasyPrint рендерит HTML в A4 PDF

## Отработка: "почему не строгая типизация?"

Короткий ответ: **мы УЖЕ используем строгую типизацию там, где она критична.**

1. **Фронтенд — TypeScript** — это строго типизированный язык. Все интерфейсы, пропсы, API-ответы типизированы. Ошибки ловятся на этапе компиляции, до запуска кода.

2. **Python + Pydantic + SQLAlchemy** — это не "динамический хаос":
   - **Pydantic** валидирует ВСЕ входящие данные на уровне API (типы, обязательные поля, форматы). Невалидный запрос не пройдёт в бизнес-логику.
   - **SQLAlchemy** строго типизирует БД-схему (Column types).
   - **FastAPI** автоматически генерирует OpenAPI-документацию из типов.
   - **130 тестов** покрывают расчётный движок — каждая формула проверена.

3. **Почему не Java/C#/Go на бэкенде:**
   - Python + FastAPI — **в 3-5 раз меньше кода** для того же результата. Для проекта нашего масштаба это означает **быструю разработку и итерации**.
   - WeasyPrint (генерация PDF) — зрелая Python-библиотека, аналогов такого уровня в Go/Java нет без значительных усилий.
   - **Скорость разработки** критичнее, чем скорость выполнения — наш бэкенд считает формулы и генерирует PDF, а не обрабатывает миллионы запросов в секунду.

4. **Если проект вырастет** — Python бэкенд легко масштабируется горизонтально (Docker контейнеры), а критичные участки можно оптимизировать точечно. Переписать на другой язык — это потерять месяцы работы без реальной выгоды на текущем этапе.

**Итого:** TypeScript на фронте + Python с Pydantic на бэкенде — это промышленный стандарт для веб-приложений такого класса. Так же работают Stripe, Dropbox, Instagram.
